1. 사이트 필수 구성 요소 (MVP)
입력 및 연동 파트

GitHub URL 입력창: 직관적이고 눈에 띄는 메인 검색창.

GitHub OAuth 로그인: 공개 저장소뿐만 아니라 사용자의 Private 저장소(개인 프로젝트, 사내 프로젝트)도 시각화할 수 있도록 권한을 위임받는 기능이 필수입니다. API 호출 제한(Rate Limit)을 우회하기 위해서라도 로그인이 필요합니다.

핵심 시각화 캔버스

드래그 앤 줌 (Pan & Zoom): 노선도가 커질 경우 마우스 휠로 확대/축소하고 드래그하여 이동하는 기능이 맵 형태의 UI에서는 가장 기본입니다.

메인 라인 고정 (Main/Master Trunk): 구상하신 대로 main 브랜치가 굵고 곧은 중앙선(지하철의 메인 호선)으로 중심을 잡고, 서브 브랜치들이 뻗어나가는 뼈대 구조를 시각적으로 확립해야 합니다.

노드(Commit) 상호작용: 커밋 노드(지하철 역)에 마우스를 올리거나 클릭했을 때 상세 정보(커밋 해시, 작성자, 날짜, 커밋 메시지, 변경된 파일 수)가 툴팁이나 사이드 패널로 표시되어야 합니다.

제어 및 필터링

브랜치 토글 스위치: 특정 브랜치(예: 수많은 feature/* 브랜치)를 끄고 켤 수 있는 체크박스.

타임라인 슬라이더: 특정 기간 동안의 커밋만 필터링해서 볼 수 있는 기능.

2. 온라인 사용자들의 주요 니즈 및 페인포인트
기존의 Git GUI 툴이나 GitHub의 기본 Network Graph를 사용하면서 유저들이 자주 불편함을 느끼고 요구하는 핵심 포인트들입니다.

스파게티 선 꼬임 방지 (가독성):

니즈: 브랜치가 많아지고 병합(Merge)이 잦아지면 선들이 거미줄처럼 얽혀서 도저히 알아볼 수 없는 상태가 됩니다.

해결책: 선의 교차를 최소화하는 스마트한 라우팅 알고리즘이 필요합니다. D3.js의 힘을 빌리거나, React 및 TypeScript 생태계의 다양한 그래프 렌더링 라이브러리를 활용하여 데이터 흐름과 노드 배치를 깔끔하게 제어하는 것이 좋습니다.

대규모 저장소 성능 문제 (렌더링 최적화):

니즈: Linux 커널이나 React 같은 거대한 오픈소스 URL을 넣었을 때, 수만 개의 커밋을 로딩하다가 브라우저가 뻗어버리는 문제가 잦습니다.

해결책: 화면에 보이는 영역만 렌더링하는 가상화(Virtualization) 기술이나, Node.js 기반의 백엔드에서 미리 커밋 데이터를 파싱하고 압축하여 프론트엔드로 전달하는 구조가 필요합니다.

포트폴리오용 이미지 추출 (Export):

니즈: 개발자들은 자신의 기여도나 프로젝트의 멋진 성장 과정을 블로그나 이력서에 넣고 싶어 합니다.

해결책: 완성된 스킬트리/지하철 맵을 고해상도 PNG나 PDF로 깔끔하게 저장할 수 있는 '내보내기' 버튼이 있으면 바이럴 마케팅에도 큰 도움이 됩니다.

게이미피케이션 및 테마 (시각적 즐거움):

니즈: 딱딱한 업무용 툴이 아닌, 보는 맛이 있는 툴을 원합니다.

해결책: 사용자가 UI 테마를 직접 고를 수 있게 하세요. 예를 들어 '런던 지하철 테마', '다크 소울 스킬트리 테마', '사이버펑크 테마' 등을 제공하면 유저들의 폭발적인 반응을 이끌어낼 수 있습니다.
